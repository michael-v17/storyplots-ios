# Seed Standard v2

> A reference guide for defining the structure of a greenfield project seed.
> Describes which files exist, what each contains, who creates them, and when.
> **Framework-agnostic**: any downstream tool (vanilla Claude Code, spec-kit, ECC, Codex, or others) can consume a seed that follows this standard.
>
> This document defines **structure and authoring flow only**. Per-framework kickoff prompts — how to wire the seed to a specific tool — are outside this scope. Each framework produces its own binding files from the seed.

---

## 1. What the Seed Standard v2 is — and isn't

**The seed** is a single `seed/` folder containing product authorship: vision, stories, schema, UX, design brief, stack, and roadmap. It also holds the reference analysis the project is based on. It is **portable**: any AI-assisted development framework can read it and execute from it.

**A greenfield v2 project has three zones:**

1. **The seed** (`seed/`) — portable product content, described in this standard
2. **Project aux files** (`README.md`, `CHANGELOG.md`, `dev-runbook.md`) — root-level files, not framework-specific
3. **Framework binding** (`CLAUDE.md` for Claude Code, or equivalent) — generated by the chosen framework's kickoff prompt; this standard does not define its contents

**The seed is NOT:** a framework, a sprint backlog, a task tracker, a runtime log, or a kickoff prompt. The seed describes **what the product is**; the framework decides **how to build it**.

---

## 2. The cornerstone: reference analysis

Most greenfield projects are based on an existing app or site — something the creator studied and wants to replicate, adapt, or build on top of. The **reference analysis** (`seed/references/[AppName]/`) is the cornerstone of the seed: a structured, in-depth breakdown of that observed app, covering its screens, flows, data model, interaction patterns, and design system.

**Why it matters:** without reference analysis, every generated file (schema, UX, user stories, roadmap) must be invented from scratch. With it, the AI extracts real behavior, real flows, and real data shapes from observed evidence — and the creator only needs to say "take this, change that, add this other thing." The reference is what makes generation grounded instead of speculative.

**What `creator-vision.md` does in this context:** it is the creator's voice shaping the entire seed. The creator can say anything here — "I'm basing this on [AppName], I want to keep X, change Y, add Z, use FastAPI, make it feel dark and minimal, and never show ads." Every generated file extracts and formalizes the relevant parts of that raw input into proper spec format. Tech preferences land in `tech-stack.md`. Design feelings land in `design.md`. Behavioral rules land in `schema.md` or `domain.md`. The creator doesn't need to know which file each thought belongs to — they just write, and generation does the sorting.

**Each generated file is produced in its own dedicated session**, using the reference + creator-vision as primary inputs. A file is not generated until all its inputs are ready.

---

## 3. Philosophy

Six principles that govern seed authorship:

1. **Omission is as dangerous as invention.** If the seed doesn't define something important, the AI will invent it incorrectly. Being explicit beats being brief.
2. **Ambiguity is a defect.** A phrase that can be interpreted two ways will be interpreted both ways across different runs. Resolve or escalate — never leave it open.
3. **One file, one purpose.** Duplicating content across files invites drift. If two files address the same topic, one is authoritative and the other links.
4. **Portable: don't pre-empt the framework.** The seed lives in a single `seed/` folder. Don't create executive subfolders (`specs/`, `plans/`, `tasks/`) that collide with downstream framework conventions.
5. **Don't assume initial perfection.** The seed can evolve with creator approval during implementation. `open-questions.md` is the pressure valve for ambiguities discovered late.
6. **Evidence-first.** Every decision in the seed cites authority: the highest-precedence file that supports it, or reference material that evidences it. Never anonymous invention.

## 4. Authoring layers

The seed is built in four layers. Understanding this prevents mixing creator input with AI-generated content.

### Layer 1 — Creator brings (already done, trusted)

These exist **before** any generation session. The creator either produced them or takes full responsibility for them. They are the raw inputs from which everything else is generated.

| File / Folder | What it is |
|---|---|
| `seed/creator-vision.md` | Creator's intent, ideas, and principles — written as a directed response to the reference: what to keep, change, and add. |
| `seed/user-stories.md` | What users need to do. Flows with acceptance criteria. Written or approved by the creator — deeply grounded in the reference. |
| `seed/references/[AppName]/` | Structured analysis of the observed app: screens, flows, data model, interaction patterns, design system. The cornerstone. |
| `design-system/` *(at root, optional)* | Concrete visual implementation (tokens, components, previews). Generated by Claude Design. Brought ready by the creator. |

> **Note on `user-stories.md`:** although the creator takes ownership, user stories are deeply grounded in the reference — the flows, screens, and behaviors observed in the reference app are the primary source. The creator's role is to confirm what to keep, cut, or add on top.

### Layer 2 — AI generates in dedicated sessions (from Layer 1 inputs)

Each file is generated in its own dedicated session. The AI reads the reference analysis and creator-vision together, extracts insights from the observed app, and applies the creator's direction. The creator reviews and approves each output before the next session begins.

| File | Primary inputs | When to include |
|---|---|---|
| `seed/schema.md` | references + creator-vision + user-stories | Always |
| `seed/ux.md` | references + user-stories + creator-vision | Only for UI projects |
| `seed/tech-stack.md` | creator-vision + schema + references | Always |
| `seed/product.md` | creator-vision + user-stories | When scope complexity warrants an explicit out-of-scope document |
| `seed/domain.md` | references + creator-vision + user-stories + schema | When there are non-obvious business invariants or complex entity lifecycles |
| `seed/architecture.md` | product + domain + tech-stack | Only when there are multiple services with real integration boundaries |
| `seed/design.md` | creator-vision + references + design-system | Only for UI projects; thin — visual principles + anti-patterns + pointer to `design-system/` |

> **Reference is the primary source for every Layer 2 file.** Schema is grounded in the data model observed in the reference. UX is grounded in the screens and flows observed. Domain invariants come from how the reference app behaves. The AI extracts, structures, and applies the creator's direction on top — it does not invent.

### Layer 3 — Roadmap in its own dedicated session (last)

The roadmap is authored **after** schema, ux, and tech-stack are complete. It needs those as inputs to sequence phases correctly. Building it earlier means speculating about content that isn't yet defined.

| File | Inputs | Notes |
|---|---|---|
| `seed/roadmap.md` | user-stories + schema + ux + tech-stack + references | One dedicated session; the most decision-heavy step |

### Layer 4 — Framework kickoff generates (outside `seed/`)

The creator chooses a downstream framework. Its kickoff prompt reads the seed and generates framework-specific operational files. For Claude Code projects, **`CLAUDE.md` is always generated** by the kickoff along with the other aux files.

**Tools that work with any framework** (orthogonal to framework choice):
- `context7` — current library/API documentation
- `playwright` — end-to-end UI verification
- web search — live information retrieval
- `serena` — semantic code navigation

**Framework-specific tools** (each framework has its own equivalents):
- Claude Code: `code-review`, `code-simplifier`, `feature-dev`, `frontend-design`
- spec-kit / ECC: their own review and generation mechanisms

**Workflow mechanisms** (declared in the framework binding, not in the seed):
- `ultraplan` — a workflow skill/prompt for Claude Code that drives plan generation. Declared in `CLAUDE.md`.

## 6. Seed files — content per file

Each entry: **purpose** · **author** · **primary inputs** · **contains** · **does NOT contain**.

### 6.1 `seed/creator-vision.md` — LAYER 1, ALWAYS

- **Purpose:** the creator's raw intent — free-form thoughts on any aspect of the product. The highest-authority file in the seed and the primary source from which all generated files are derived.
- **Author:** creator — written by hand or dictated conversationally. No required format or structure.
- **Primary inputs:** none — but written informed by `references/[AppName]/` and the creator's own ideas.
- **Contains:**
  - What the creator is basing this on (the reference app) and why
  - What to keep from the reference, what to change, what ideas to add on top
  - Opinions on tech stack ("I want FastAPI", "no Docker for now", "keep it simple")
  - Design and feel preferences ("dark, intimate, like Notion", "no bright colors")
  - Behavior and interaction preferences ("chat should stream", "editing a message resets history")
  - Non-negotiable principles — invariants that must never be violated
  - Prioritization principles under ambiguity (what weighs more when there's tension)
  - Tone, voice, or any other thought the creator considers important
- **Format:** free-form prose, lists, fragments — whatever the creator naturally writes. This is intentionally unstructured. The generated files formalize these raw thoughts into their proper spec format.
- **Does NOT contain:** formal spec structures (schema tables, screen inventories, migration scripts). Those are what the AI produces by formalizing the creator's informal input. The creator describes intent; the generated files give it structure.

### 6.2 `seed/user-stories.md` — LAYER 1, ALWAYS

- **Purpose:** what users need to do. Critical flows (F1..FN) with acceptance criteria and priority.
- **Author:** creator — written, approved, or co-drafted conversationally. Heavily grounded in the reference.
- **Primary inputs:** reference analysis + creator-vision.
- **Contains:**
  - Stories grouped by area or flow. Each with: Title, Priority (Critical / High / Medium / Low), As-a / I-want / So-that, Why it matters, Acceptance criteria, Related screens, Related entities, Notes
  - Non-negotiable flows enumerated as F1..FN with sequential steps
  - Explicit priority for each story
- **Does NOT contain:** implementation, exact layouts, physical schema.

### 6.3 `seed/references/[AppName]/` — LAYER 1, OPTIONAL

- **Purpose:** structured analysis of the observed app. The cornerstone of the seed — the primary evidence base for all generated files.
- **Author:** creator produces the analysis (separately, before the seed generation sessions). Brought ready.
- **Naming:** parent folder `references/` lowercase inside `seed/`. App subfolder named after the observed app in its natural casing (`PersonaLLM/`, `Twitter/`, `Notion/`). If there is more than one reference app, each gets its own subfolder.
- **Recommended internal structure:**
  ```
  [AppName]/
    00-index.md                     — overview + file map of the analysis
    01-overview.md                  — what the app is, who it serves, core value
    02-information-architecture.md  — navigation, sections, hierarchy
    03-data-model.md                — entities, relationships, observed schema
    04-screens/                     — per-screen breakdowns
    05-flows.md                     — critical user flows step by step
    06-interaction-model.md         — how the app behaves (states, gestures, patterns)
    07-technical-notes.md           — observed constraints, API patterns, performance
    08-design-system.md             — visual language, typography, spacing, components
    ...                             — add files as needed
  ```
- **Does NOT contain:** anything authoritative over the seed. If the reference suggests something that conflicts with `creator-vision.md`, creator-vision wins. Conflicts go to `open-questions.md`.

> **Why this structure matters:** the more structured and thorough the reference analysis, the less the AI needs to invent. Schema, UX flows, domain invariants, design language — all grounded in evidence rather than speculation. A weak or missing reference means more open questions and more invention risk.

### 6.4 `seed/schema.md` — LAYER 2, ALWAYS

- **Purpose:** concrete data shape + isolation rules + business invariants + canonical terminology.
- **Author:** AI generates; creator reviews.
- **Primary inputs:** references + creator-vision + user-stories.
- **Contains:**
  - Tables with fields + types
  - Relationships (FK), unique constraints, relevant indexes
  - RLS / scoping rules per entity (what is isolated by user / conversation / etc.)
  - Storage boundary rules (what must not bleed across contexts)
  - **Business Rules / Invariants section** — behavioral rules that can never be false (e.g., "editing a message trims the conversation from that point forward")
  - **Glossary section** — canonical names for entities and concepts
- **Does NOT contain:** domain prose (if `domain.md` exists, that lives there), migrations (the framework creates them), UI.

### 6.5 `seed/tech-stack.md` — LAYER 2, ALWAYS

- **Purpose:** chosen stack with rationale + explicit prohibitions + layer responsibilities + testing stack.
- **Author:** creator + AI decide together; creator approves pinned versions.
- **Primary inputs:** creator-vision + schema + references (for observed stack clues).
- **Contains:**
  - Layers table: language / runtime / framework / DB / testing / tooling — with choice, version, rationale
  - Pinned versions (no `^` or `~` ranges on main deps)
  - **"What we are NOT using"** — explicit prohibitions (e.g., "No ORM", "No Docker yet", "No Redux")
  - **Testing stack** — unit test runner, e2e tool, and what each covers (e.g., "vitest for unit, playwright for e2e smoke")
  - Layer responsibilities (what frontend does, what backend does, what agents do) — absorbs basic architecture content for single-service projects
  - **Dev services table** — per service: exact start command, port, health check command, log location. This is the source the kickoff reads to generate `dev-runbook.md`. Without it, the runbook is generic and the AI cannot manage the dev environment autonomously.
- **Does NOT contain:** run commands outside of the Dev services table (those go in `dev-runbook.md`), domain concepts, screen specs.

### 6.6 `seed/ux.md` — LAYER 2, IF UI PROJECT

- **Purpose:** screens and end-to-end flows contract.
- **Author:** AI generates; creator reviews.
- **Primary inputs:** references (screens + flows) + user-stories + creator-vision.
- **Contains:**
  - Sitemap / route map
  - Screen inventory: for each screen → route, purpose, entry points, must-have sections, optional sections, primary actions, secondary actions, interactions, what it opens, required states (empty / loading / error / success), critical edge cases, must-not-omit
  - Modal / overlay registry
  - Critical flows end-to-end (step by step)
  - Non-omission rules (what cannot be missing from a screen)
- **Does NOT contain:** visual tokens (live in `design-system/`), backend behavior, schema.

### 6.7 `seed/roadmap.md` — LAYER 3, ALWAYS · dedicated session last

- **Purpose:** phase sequence with scope + exit criteria per phase. The authoritative answer to "what do we build, and in what order?"
- **Author:** creator + AI in a dedicated session, after all Layer 2 files are complete.
- **Primary inputs:** user-stories + schema + ux + tech-stack + references.
- **Contains:**
  - Phases 0..N in strict order. Each phase with:
    - Scope bullets (what it delivers)
    - User stories covered (by ID from user-stories.md)
    - Flows covered (F1..FN — full or partial)
    - Explicit out-of-scope (what is NOT touched in this phase — and which phase covers it)
    - **Objective exit criteria — written as executable commands**, not narratives:
      ```
      - pnpm typecheck → 0 errors
      - vitest run → all pass
      - playwright: [specific flow] → [expected result]
      - curl localhost:8000/health → {"status":"ok"}
      ```
    - Applicable non-negotiables check
  - Status per phase: `⏳ pending` / `🔒 ready to start` / `✅ shipped`
  - **"Out of roadmap"** section — features discarded or deferred indefinitely
- **Does NOT contain:** atomic per-phase tasks (the framework instantiates those when a phase begins), concrete stack (comes from tech-stack.md).

> **Why executable exit criteria matter:** without objective gates, "done" defaults to "the creator approves" — which puts the creator in the loop for every phase. Executable criteria let Claude verify completion autonomously: run the commands, check the results, mark ✅ or fix. This is what enables low-intervention development in vanilla Claude Code without a framework-enforced validation file.

> **Why last:** the roadmap needs to know what's being built (schema, UX) and how (stack) to sequence phases correctly. One dedicated session with all inputs ready produces a roadmap that stays stable throughout the project.

### 6.8 `seed/open-questions.md` — OPERATIONAL · append-only

- **Purpose:** register of ambiguity detected during authorship and implementation.
- **Author:** AI adds when it detects a high-impact gap; creator resolves when possible.
- **Contains:**
  - Open questions grouped by area (product, domain, schema, ux, design, stack, roadmap)
  - Each with: brief context, why it matters, proposed default if any, status (open / resolved)
  - Contradictions detected between seed files
  - Temporary assumptions not yet confirmed
- **Does NOT contain:** final answers — when resolved, the decision moves to the appropriate seed file; this file keeps only `RESOLVED: moved to <file>`. Low-impact or cosmetic questions don't belong here.

### 6.9 `seed/product.md` — LAYER 2, CONDITIONAL

Include when: the project's scope is complex enough that an explicit out-of-scope document adds real clarity. For simpler projects, out-of-scope content lives in `roadmap.md` under "Out of roadmap."

- **Purpose:** product definition — scope, MVP cutline, explicit out-of-scope, success criteria.
- **Primary inputs:** creator-vision + user-stories + references.
- **Contains:** problem statement, target users, key user outcomes, MVP scope, explicit out-of-scope, observable success criteria, high-level constraints.
- **Does NOT contain:** tech stack, screen details, schema.

### 6.10 `seed/domain.md` — LAYER 2, CONDITIONAL

Include when: there are non-obvious business invariants, complex entity lifecycles, or terminology requiring explicit definition. For simpler projects, invariants live as a "Business Rules" section in `schema.md` and terminology as a "Glossary" section there.

- **Purpose:** conceptual domain model — the rules and vocabulary of the business.
- **Primary inputs:** references + creator-vision + user-stories + schema.
- **Contains:** entity hierarchy and relationships, lifecycle of important objects, ownership boundaries, conceptual invariants, canonical terminology and glossary.
- **Does NOT contain:** physical schema, UI, concrete backend behavior.

### 6.11 `seed/architecture.md` — LAYER 2, CONDITIONAL

Include when: the project has more than one service with real integration boundaries. For single-service projects, layer responsibilities live in `tech-stack.md`.

- **Purpose:** technical responsibilities of subsystems + cross-subsystem technical flows.
- **Primary inputs:** product + domain + tech-stack.
- **Contains:** subsystems and their responsibilities, integration boundaries (what talks to what, via what protocol), cross-subsystem technical flows (e.g., SSE streaming, background jobs), durable technical constraints.
- **Does NOT contain:** concrete stack (lives in tech-stack.md), physical schema (lives in schema.md), conceptual domain (lives in domain.md).

### 6.12 `seed/design.md` — LAYER 2, CONDITIONAL · thin

Include when: the project has UI and a `design-system/` exists. Intentionally thin — the `design-system/` is authoritative for tokens and components; this file is the brief and the guardrails.

- **Purpose:** visual brief + anti-patterns + pointer to design-system.
- **Primary inputs:** creator-vision + references + design-system.
- **Contains:** visual north star (adjectives, analogies — no hex values), design principles (2–4 lines each), explicit anti-patterns to avoid, pointer to `design-system/` as authoritative for tokens and components.
- **Does NOT contain:** concrete tokens (live in design-system/), screen specs (live in ux.md).

## 7. Files outside `seed/` — what they are and who produces them

### 7.1 `design-system/` — LAYER 1 OPTIONAL · creator brings ready

- **Purpose:** concrete visual implementation (tokens + components + previews).
- **Author:** generated by Claude Design (or equivalent); creator brings it ready before generation sessions begin.
- **Relationship to seed:** `seed/design.md` is the brief → `design-system/` is the implementation. The framework treats `design-system/` as authoritative for visual tokens and component shapes.
- **Contains:** `tokens.css` with concrete hex, rem, radii, spacing; `preview/` with rendered component HTML; `ui-kits/` with reference JSX (optional); custom fonts if applicable.
- **Does NOT contain:** visual principles (live in `seed/design.md`), screen specs (live in `seed/ux.md`).

### 7.2 `dev-runbook.md` — ROOT · generated by kickoff

- **Purpose:** how to start, verify, and restart all development services. The agent reads this at the start of each session and uses it to manage the dev environment autonomously.
- **Author:** generated by the framework kickoff prompt, based on `seed/tech-stack.md`.
- **Contains:**
  - Table of services: name / start command / port / health check command
  - Step-by-step: how the agent starts the full stack at session start
  - How to restart a service (kill + re-run)
  - How to clear stuck ports from a previous session
  - What the agent does NOT manage (e.g., cloud DB setup, `.env` secrets, production deploy)
  - Rule: all services run in background — never foreground (blocks the conversation)
- **Does NOT contain:** anything from the seed (domain, schema, UX). It is purely operational.

> **For Claude Code specifically:** the agent uses `Bash(cmd, run_in_background=true)` to start services, `BashOutput(id)` to read logs, and `KillShell(id)` to stop them. This file tells the agent exactly which commands to use for this project. Without it, the creator has to start/restart servers manually — which slows every feedback cycle significantly.

### 7.3 `README.md` — PROJECT

- **Purpose:** human onboarding to the repo.
- **Contains:** what the project is (1 paragraph), link to `seed/` as source of truth, how to start development (points to `dev-runbook.md` and `seed/tech-stack.md`).

### 7.4 `CHANGELOG.md` — RUNTIME

- **Purpose:** immutable log of what has shipped.
- **Contains:** sections by date (`## YYYY-MM-DD`), reverse-chronological, one line per commit.

### 7.5 `CLAUDE.md` (or framework equivalent) — FRAMEWORK · always for Claude Code

- **Purpose:** harness conventions for the chosen downstream framework.
- **Author:** the framework's kickoff prompt. **Always generated for Claude Code projects.**
- **Contains:**
  - Reference to seed precedence (which file wins on conflict)
  - Framework-specific execution rules and workflow mechanisms (e.g., how `ultraplan` is used)
  - Pointer to `dev-runbook.md` for dev environment management
  - Tool/plugin usage rules — distinguishing universal tools (context7, playwright, web search, serena) from framework-specific ones (code-review, code-simplifier, feature-dev, frontend-design for Claude Code)
  - Verification ritual: "before marking a phase ✅ shipped, run all exit criteria from `seed/roadmap.md` for that phase"
- **Does NOT contain:** product spec content (lives in `seed/`), run commands (lives in `dev-runbook.md`).

> **Framework binding names by tool:** Claude Code → `CLAUDE.md`, Codex → `AGENTS.md`, Cursor → `.cursorrules`, spec-kit → `memory/constitution.md`.

### 7.6 Session continuity artifact — FRAMEWORK · generated during work

Each framework produces its own version of "where we are":

| Framework | Artifact | Role |
|---|---|---|
| Claude Code | `SESSION_HANDOFF.md` (slim, <100 lines) | WIP between sessions: last commit, in-progress tasks, blockers, next step |
| spec-kit | Status in `validation.md` of the active phase | Completion state of the current phase spec |
| ECC | ECC memory system | Framework-internal continuity |

Not part of the seed. Generated and maintained per framework conventions.

### 7.7 Per-phase work artifacts — FRAMEWORK · generated per phase

When work on a phase begins, the framework generates its own execution artifacts from `seed/roadmap.md`:

| Framework | Per-phase artifacts |
|---|---|
| Claude Code | `plans/NNNN-slug.md` — one file per cycle |
| spec-kit | `specs/YYYY-MM-DD-feature/{requirements, plan, validation}.md` |
| ECC | Framework-internal |

`seed/roadmap.md` is the source of truth for what phases exist and when they're done. The framework artifacts are how each framework executes within a phase.

## 8. Document precedence (within `seed/`)

Authority hierarchy within `seed/`, highest to lowest:

1. `seed/creator-vision.md` — **absolute authority**
2. `seed/user-stories.md`
3. `seed/product.md` *(if present)*
4. `seed/domain.md` *(if present)*
5. `seed/schema.md`
6. `seed/ux.md` *(if present)*
7. `seed/design.md` *(if present)*
8. `seed/tech-stack.md`
9. `seed/roadmap.md`
10. `seed/architecture.md` *(if present)*
11. `seed/open-questions.md`
12. `seed/references/[AppName]/` — **never authoritative; always interpretive**

**Resolution rules:**
- If a lower-authority file contradicts a higher-authority one, the higher one wins.
- If the contradiction is material (changes system meaning), surface it to the creator.
- If unresolvable, it goes to `open-questions.md`.
- `references/` never wins over any seed file. It informs; the seed decides.

**Outside `seed/`:** `README.md`, `CHANGELOG.md`, `dev-runbook.md`, and the framework binding do NOT participate in this hierarchy. If they contradict the seed, they are wrong and must be corrected.

## 9. Minimum viable seed

Five files are sufficient for any framework to start. Everything else improves quality but does not block the start.

**The five spark files:**

| File | What it provides |
|---|---|
| `creator-vision.md` | Why + principles + non-negotiables |
| `user-stories.md` | What users need + critical flows |
| `schema.md` | Data model + isolation rules + invariants |
| `ux.md` | Screens + flows *(omit if no UI)* |
| `roadmap.md` | Phase sequence + executable exit criteria |

A framework reading only these five files can determine: what to build, for whom, what the data looks like, how the UI is structured, and in what order to build it.

**Conditional enrichers** (add when the project warrants):
- `tech-stack.md` — always add; sometimes authored in its own session after the others
- `product.md` — add when out-of-scope requires an explicit document
- `domain.md` — add when business rules are non-obvious
- `architecture.md` — add when multi-service
- `design.md` — add when UI + design-system exists

## 10. Portability contract

For the seed to be framework-agnostic, it must satisfy these **six universal conditions**:

1. **Plain markdown** — no framework-specific template syntax, macros, or include directives
2. **Single `seed/` folder** — no executive subfolders (`specs/`, `plans/`, `tasks/`) that collide with downstream framework conventions
3. **Explicitly declared precedence** (§8) — the framework knows which file wins without heuristics
4. **Consistent naming** (§11) — the framework can parse the shape without special cases
5. **No slash commands or plugin dependencies** — content readable by any LLM, not one in particular
6. **Universal guardrails: non-invention + non-omission + non-ambiguity** — apply to any framework and any agent

## 11. Naming conventions

| Rule | Applies to | Reason |
|---|---|---|
| `seed/` lowercase | Main seed folder | Filesystem standard |
| `kebab-case.md` | Files inside `seed/` | Readable, standard markdown on GitHub |
| `references/` lowercase | Reference container folder inside `seed/` | Consistent with seed/ convention |
| `[AppName]/` natural casing | App reference subfolder | Preserves the app's identity (e.g., `PersonaLLM/`, `Twitter/`, `Notion/`) |
| `UPPERCASE.md` at root | `README.md`, `CHANGELOG.md`, `CLAUDE.md` (and equivalents) | GitHub + Claude Code convention; high visibility |
| `dev-runbook.md` lowercase at root | Dev runbook | Operational file; lowercase consistent with other non-spec root files |
| `lowercase/` at root | `design-system/` | Optional folder; filesystem standard |

## 12. Authoring rules

### Inherited (proven from v1)

**Rule 1 — creator-vision priority.** `creator-vision.md` has absolute authority. If it contradicts another file, it wins.

**Rule 2 — user-story authority.** `user-stories.md` has high authority over ux, domain, and architecture. It represents what users actually need in practice.

**Rule 3 — input-first generation.** The reference analysis and creator-vision are the primary inputs. The rest of the seed is generated from them, not from generic assumptions.

**Rule 4 — reference-awareness.** `references/[AppName]/` is interpretive input — use it actively and deeply, but never treat it as authoritative. The seed decides; the reference informs.

**Rule 5 — non-invention.** If something important is not clearly defined, don't invent it. Identify the gap and record it in `open-questions.md`.

**Rule 6 — non-omission.** Do not omit required screens, states, flows, entities, schema boundaries, creator principles, or high-priority user outcomes.

**Rule 7 — explicit priority.** The seed must make explicit what is critical / high / secondary / unresolved. No ambiguity.

**Rule 8 — hybrid format.** Use prose where interpretation matters; tables and lists where precision matters.

**Rule 9 — alignment.** All seed files must align with each other. A lower-precedence file cannot drift from a higher one.

**Rule 10 — contradiction escalation.** If `creator-vision.md` contradicts another file, don't reconcile silently. Creator-vision wins; if the change of meaning is material, surface it to the creator.

**Rule 11 — reference conflict.** If `references/` suggests something that contradicts the seed, don't silently merge it. Preserve seed precedence; if material, go to `open-questions.md`.

**Rule 12 — review-aware.** Assume the creator will review the seed. Optimize for legibility and inspectability. Make important assumptions visible.

### New for v2

**Rule 13 — single-folder.** The seed lives in `seed/` as the only folder. Don't create executive subfolders that collide with downstream framework conventions.

**Rule 14 — non-duplication.** A fact lives in exactly one file. If two files need the same content, one is authoritative and the other links or cites.

**Rule 15 — portability.** The seed does not reference slash commands, plugins, workflow skills, or conventions specific to any one framework. Operational content stays outside the seed.

**Rule 16 — evolution.** The seed can be updated during implementation when foundational decisions change. Requires creator approval and a version note in `creator-vision.md`. Mostly stable, not frozen absolutely.

**Rule 17 — phasing.** `tech-stack.md` and `roadmap.md` may be authored in later sessions. They are not optional — they are late by necessity. The seed is incomplete until they exist but valid as an operative subset.

**Rule 18 — layer discipline.** `creator-vision.md` is the creator's raw voice — free-form, any topic, no required structure. Generated files are the formal derivation of that raw input. Don't inject AI-synthesized structure into `creator-vision.md`, and don't expect it to be a spec. Its authority comes from being the creator's authentic intent, not from being well-structured.

**Rule 19 — roadmap last.** The roadmap is always the last seed file generated. It needs schema, ux, and tech-stack as inputs to sequence phases correctly.

**Rule 20 — one session per file.** Each Layer 2 file is generated in its own dedicated session. Don't generate multiple files in one pass — each deserves focused attention on its specific inputs.

**Rule 21 — executable exit criteria.** Every phase in `roadmap.md` must have exit criteria written as runnable commands (typecheck, test runner, curl, playwright assertion) — not narrative descriptions. This is what allows Claude to verify completion without creator approval at every step.

**Rule 22 — dev runbook in kickoff.** The `dev-runbook.md` is generated by the framework kickoff, not authored by the creator. It depends on knowing the full tech stack. For Claude Code: servers always run in background; Claude reads its own logs; three failed restarts = escalate.

## 13. Authoring flow — what exists when

**State 0 — Pre-seed. Creator provides:**
- `seed/creator-vision.md`
- `seed/user-stories.md`
- `seed/references/[AppName]/` (very common in greenfield — the cornerstone)
- `design-system/` at root (if UI project — generated by Claude Design)

**State 1 — Core generation (one dedicated session per file).**
AI generates from State 0 inputs; creator reviews and approves each before the next:
- `seed/schema.md`
- `seed/ux.md` (if UI)
- `seed/tech-stack.md` (includes testing stack + Dev services table)
- `seed/product.md` (if scope warrants)
- `seed/domain.md` (if domain complexity warrants)
- `seed/architecture.md` (if multi-service)
- `seed/design.md` (if UI + design-system)
- `seed/open-questions.md` (emerges during the process)

Use `generation-guides/` (in the SeedKit folder) for each file: each guide contains the session opening prompt, the extraction map, the required output structure, quality gates, and common failure modes for that specific file. One guide = one dedicated session.

**State 2 — Roadmap session (dedicated, last).**
With all State 1 files complete, creator + AI author in a focused session:
- `seed/roadmap.md` (with executable exit criteria per phase)

Use `generation-guides/08-roadmap.md` for this session.

→ **Seed is complete.**

**State 3 — Framework kickoff.**
Creator chooses a framework (or stays with Claude Code vanilla). The kickoff reads the seed and generates:
- `CLAUDE.md` (always for Claude Code) or framework equivalent
- `dev-runbook.md` — generated from `seed/tech-stack.md` §Dev services; exact commands for this project's stack. Use `generation-guides/09-dev-runbook.md` for this step.
- `README.md` at root (if not present)
- `CHANGELOG.md` (empty, ready for auto-generation)
- Session continuity artifact (per framework conventions)

**State 4 — Implementation begins.**
The framework reads `seed/` as source of truth and starts Phase 0 of the roadmap. It creates its own per-phase artifacts per its conventions (`plans/` for Claude Code, `specs/` for spec-kit). The seed stays read-mostly; updated only with creator approval when a foundational decision changes (Rule 16).

The agent verifies each phase by running the exit criteria from `seed/roadmap.md` before marking it ✅ shipped — no creator approval required if all criteria pass.

## 14. Quality standard

**A strong seed:**
- Preserves the creator's intent unmistakably
- Is grounded in reference evidence — not invented from nothing
- Defines scope clearly (what's in, what's out)
- Explains what users need to do (stories + flows + priority)
- Establishes schema with explicit isolation rules and invariants
- Prevents incomplete screens / states / flows via non-omission rules
- Has executable exit criteria per phase — not narrative gates
- Reduces structural hallucination by the AI
- Satisfies the 6 portability conditions
- Each file has one clear purpose with no redundancy

**A weak seed:**
- Vague or generic — could be any product
- Shallow — surface-level only
- Contradictory between files
- Invented without reference grounding
- Full of mood but lacking real UX coverage
- No scoping, isolation rules, or explicit out-of-scope
- Creator's voice mixed with AI synthesis in `creator-vision.md`
- Files that duplicate content or have no distinct role
- Exit criteria written as narratives, not runnable commands
- Missing roadmap — leaves sequencing decisions to the AI at runtime

---

## 15. What comes after the seed

With the seed complete (States 0–2), you have a **portable, framework-agnostic, evidence-grounded artifact** that describes WHAT the product is and in what order to build it.

The next step is the framework kickoff (State 3), which produces the operational layer: `CLAUDE.md`, `dev-runbook.md`, and the session continuity artifact. From that point, the framework executes the roadmap phase by phase.

**The seed is not rewritten** during implementation — only with creator approval when a foundational decision changes (Rule 16). The framework **consumes** the seed; it does not rewrite it.

When implementation is complete, the seed remains as a permanent record of what the product was when it was built — reusable to re-implement with another framework, or to onboard a new collaborator months later.
