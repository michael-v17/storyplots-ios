# Generation Guide: seed/ux.md

## 1. When to include

**Always in UI projects.** Omit only if the project has no user interface (API-only, CLI, etc.).

It is one of the 5 spark files. Without `ux.md`, the AI can implement the schema and the backend but has no contract specifying which screens to build, in what order, or which states are mandatory.

---

## 2. Prerequisites

These must exist before opening this session:

- [ ] `seed/references/[AppName]/04-screens/` — per-screen breakdowns of the reference app
- [ ] `seed/references/[AppName]/05-flows.md` — critical flows observed step by step
- [ ] `seed/references/[AppName]/02-information-architecture.md` — observed navigation structure
- [ ] `seed/user-stories.md` — flows F1..FN with acceptance criteria
- [ ] `seed/creator-vision.md` — what is removed, what is added, required breakpoints

`schema.md` is not a strict prerequisite, but if it already exists it helps verify that the screens reference the correct entities.

---

## 3. Session opening prompt

```
We are going to generate seed/ux.md. This is the screens and flows contract — it defines
which screens exist, what they must contain, and how the critical flows execute
from start to finish.

Read the following files in this order before starting:
1. seed/references/[AppName]/02-information-architecture.md  — observed navigation
2. seed/references/[AppName]/04-screens/                     — breakdown of each screen
3. seed/references/[AppName]/05-flows.md                     — critical flows step by step
4. seed/user-stories.md                                      — mandatory flows F1..FN
5. seed/creator-vision.md                                    — what is removed, what is added

Produce seed/ux.md with this structure:
- Sitemap: route map for the project
- Navigation model: how navigation works (sidebar, bottom nav, top bar) + what was removed vs reference
- Breakpoints: mobile (S), tablet (M), desktop (L) with their layout pattern
- Screen contracts: for each screen → route, purpose, entry points, must-have sections,
  optional sections, primary/secondary actions, interactions, what it opens, required states,
  critical edge cases, must-not-omit
- Modal/overlay registry: list of all modals and overlays with their trigger and content
- Critical flows end-to-end: each flow F1..FN from the seed step by step
- Non-omission rules: what cannot be absent in any implementation

Rules:
- NEVER invent screens that are neither in the reference nor in creator-vision
- Each flow F1..FN from user-stories MUST have its section in the end-to-end flows
- Required states are a minimum of 4 per screen: empty, loading, error, success/populated
- Explicitly document what was removed vs the reference and why
- Cite the source for each screen: (observed: reference) or (new: creator-vision §X)
```

## 4. Extraction map

### From `seed/references/[AppName]/04-screens/`
- Complete list of screens from the reference (names + routes)
- Per screen: main purpose, observed sections, available actions, observed states
- Screens the creator has indicated should be removed (cross-reference with creator-vision)
- Reference-specific layout patterns (sidebar on desktop, bottom nav on mobile, etc.)

### From `seed/references/[AppName]/05-flows.md`
- Observed step-by-step flows → base for the seed's critical flows
- Entry and exit points of each flow
- Observed edge cases (what happens if the flow fails midway)

### From `seed/references/[AppName]/02-information-architecture.md`
- Main navigation structure (what is the first-level nav)
- Screen hierarchy (what is nested inside what)
- Observed navigation gestures or shortcuts

### From `seed/user-stories.md`
- §6 flows F1..FN — these are the flows that MUST appear in ux.md as "Critical flows" sections
- Related screens of each story → verify that every referenced screen exists in the sitemap
- Acceptance criteria → many map to "required states" or "must-not-omit" items for screens

### From `seed/creator-vision.md`
- Screens or features removed vs the reference (with rationale)
- New screens or features the reference does not have
- Breakpoint preferences or web adaptation notes
- Any navigation options removed (e.g. "no bottom navigation on mobile — use collapsible sidebar")

## 5. Required output structure

### 5.1 Sitemap
Route map in table or ASCII format:

```
/                     → Home (list of conversations)
/characters           → Characters list
/characters/new       → New character form
/characters/:id/edit  → Edit character
/chat/:conversationId → Chat view
/settings             → Settings hub
/settings/text-engine → Text engine settings
...
```

### 5.2 Navigation model

- Describes the navigation pattern per breakpoint (S/M/L)
- Explicit list of what exists vs the reference (with rationale for removals)

```markdown
## Navigation model

**Desktop (L):** Persistent sidebar with: [items]. No bottom navigation.
**Mobile (S):** Collapsible sidebar (hamburger). No bottom navigation.

**Removed vs reference:**
- Bottom navigation → removed; StoryPlots web uses sidebar at all breakpoints
  (creator-vision §3: web-first, sidebar as primary nav)
- Community section → removed; out of v0 scope (creator-vision §1 anti-goals)
```

### 5.3 Breakpoints

| Breakpoint | Width range | Layout pattern | Navigation pattern |
|---|---|---|---|
| S (mobile) | < 768px | Single column | Collapsible sidebar |
| M (tablet) | 768px–1024px | [pattern] | [pattern] |
| L (desktop) | > 1024px | Sidebar + content | Persistent sidebar |

### 5.4 Screen contracts

**The most extensive section of the document.** For each screen:

```markdown
### [Screen Name]

**Route:** `/path/:param`
**Purpose:** what the user does here (1-2 lines)
**Entry points:** how you get here (from which screens or actions)
**Source:** (observed: reference) | (new: creator-vision §X)

**Must-have sections:**
- [Section A]: exactly what it contains
- [Section B]: exactly what it contains

**Optional sections:**
- [Section C]: under what condition it appears

**Primary actions:** the main action the user takes here
**Secondary actions:** secondary actions available
**Interactions:** gestures, hover states, inline edits, etc.
**Opens:** which screens, modals, or sheets it opens

**Required states:**
- Empty: what is shown when there is no data (not a blank canvas — instruction or CTA)
- Loading: indicator while loading
- Error: how the error is handled (message, retry, etc.)
- Populated/Success: normal state with data

**Critical edge cases:**
- [Edge case 1]: what happens and how it is handled
- [Edge case 2]: ...

**Must-not-omit:**
- [Element A] — reason it cannot be missing
- [Element B] — reason it cannot be missing
```

**Note**: "Must-not-omit" is the section that prevents the implementer from silently reducing the complexity of the screen. Include here everything that the AI or a developer might accidentally omit.

### 5.5 Modal / overlay registry

Table of all modals, sheets, drawers, and overlays in the system:

| Modal ID | Trigger | Content | Actions | Returns to |
|---|---|---|---|---|
| new-character-modal | Click "New character" from Characters list | Character creation form | Save / Cancel | Characters list (refreshed) |
| ... | ... | ... | ... | ... |

### 5.6 Critical flows end-to-end

One section per each flow F1..FN from `user-stories.md`. Format:

```markdown
### F2 — Send a message and receive a streaming reply

**Precondition:** User has at least one Conversation open
**Trigger:** User types a message and presses Send

Steps:
1. User types message in composer [ChatView > Composer]
2. Message appears immediately in the feed (optimistic UI) [ChatView > MessageFeed]
3. Loading indicator appears below last message [ChatView > MessageFeed]
4. Agent reply streams in token by token [ChatView > MessageFeed > AgentBubble]
5. Streaming completes; loading indicator disappears

**Error path:** If the API call fails at step 3, the message is marked with error state
and a "Retry" button appears.

**Postcondition:** Both messages are persisted; conversation scroll is at the bottom
```

### 5.7 Non-omission rules

List of globally applicable rules — things no screen can silently omit:

- Every screen with a list has an empty state with an instruction (not a blank canvas)
- Every destructive operation has a confirmation (at least an "Are you sure?" or equivalent)
- Error states always show an actionable message (not just "Error")
- [Other project-specific rules]

## 6. Quality gates before approval

- [ ] Each flow F1..FN from `user-stories.md` has its section in "Critical flows end-to-end"
- [ ] Each screen in the sitemap has its complete screen contract
- [ ] Each screen has the 4 required states documented (empty, loading, error, populated)
- [ ] The modal registry covers all modals mentioned in user-stories or in screen contracts
- [ ] The "Removed vs reference" section exists in Navigation model
- [ ] No screen was invented (each one has its source cited)
- [ ] Non-omission rules exist and are verifiable
- [ ] Entity names in ux.md match those in schema.md (if schema exists)

---

## 7. Common failure modes

**1. Inventing screens without evidence**
The AI adds an "Analytics dashboard" or "Notification center" screen that nobody requested. Detect: every screen must have `Source: (observed: reference)` or `Source: (new: creator-vision §X)` — if it has no source, it is an invention. Solution: remove it or move it to `open-questions.md`.

**2. Documenting only the happy path**
The AI writes screen contracts without the empty/loading/error states. Consequence: the implementer does not know what to show when there is no data or when a request fails. Detect: look for "Required states" in each screen contract — if missing or with fewer than 3 states, it is incomplete. Solution: complete the states before approving.

**3. Not including non-omission rules**
The document describes the screens but does not say what is untouchable. Consequence: in simplification cycles, the AI may remove "optional" sections that are actually critical. Detect: verify that the "Non-omission rules" section exists. Solution: add it before approving, consulting creator-vision §8 and user-stories ACs.

**4. Flows that do not match user-stories**
The AI describes the flows observed in the reference but omits or modifies the F1..FN flows from the seed that the creator approved. Detect: compare the F1..FN list in user-stories §6 against the list of flows in ux.md — they must match one-to-one. Solution: add the missing flows.

**5. Incomplete modal registry**
The AI documents the obvious modals (delete confirmation) but omits secondary modals (settings drawers, complex tooltips, media uploaders). Detect: search all screen contracts for any "opens" mention that is not in the modal registry. Solution: complete the registry before approving.
