## Starting a new session — READ THIS FIRST

1. **Run preflight first.** `./preflight.ps1` (Windows) or `./preflight.sh` (macOS/Linux/WSL)
   from the repo root. It verifies pinned tools are on PATH, `.env.local` parity with
   `.env.example`, opaque-key shape, and live reachability of external services — all
   without printing any secret values. Expected: `HTTP 200` on the reachability line. If it
   fails, fix the failure before coding. See `SeedKit-v2/approaches/claude-code/preflight-pattern.md`
   for the rationale and the drift-handling workflow.
2. **Read `SESSION_HANDOFF.md`** — the living project state. It tells you what has shipped,
   which phase you're in, which subtask is in progress, and the exact next step.
3. **Check `.serena/memories/`** — six canonical memories summarise project / tech-stack /
   commands / style / task-completion / structure. Any additional `decision_*_YYYY_MM_DD`
   memories describe non-obvious trade-offs you may need before editing specific areas
   (see §Mid-phase decision memories).
4. Open the current `plans/NNNN-phase-slug.md` to see the full phase detail.
5. Check `dev-runbook.md` and start any services that aren't running.

## Source of truth

The `seed/` folder is the sole authoritative source for what to build.

Document precedence within `seed/` (higher wins on conflict):

1. `seed/creator-vision.md` — absolute authority; creator intent + non-negotiables
2. `seed/user-stories.md` — what users must do; critical flows with acceptance criteria
3. `seed/product.md` *(if present)* — scope, MVP cutline, explicit out-of-scope
4. `seed/domain.md` *(if present)* — domain invariants, entity lifecycles, canonical terminology
5. `seed/schema.md` — data model, isolation rules, business invariants, glossary
6. `seed/ux.md` *(if present)* — screens, flows, required states, non-omission rules
7. `seed/design.md` *(if present)* — visual brief and anti-patterns; points to design-system/
8. `seed/tech-stack.md` — stack, prohibitions, layer responsibilities, testing stack
9. `seed/roadmap.md` — phase sequence with executable exit criteria
10. `seed/architecture.md` *(if present)* — subsystem responsibilities, integration boundaries
11. `seed/open-questions.md` — append-only; conflicts go here, not to code
12. `seed/references/[AppName]/` — interpretive only; never authoritative over seed files

Outside `seed/`: `CLAUDE.md`, `SESSION_HANDOFF.md`, `dev-runbook.md`, `plans/`, `CHANGELOG.md`
do NOT participate in this hierarchy. If they contradict the seed, they are wrong.

## Dev environment

Read `dev-runbook.md` at the start of every session and verify services are running
before writing or testing code.

Claude Code rules:
- All services start via `Bash(run_in_background=true)`. Never foreground (blocks the conversation).
- Read logs via `BashOutput(id)`.
- Kill stuck processes via `KillShell(id)`.
- After 3 failed restart attempts on a service: stop and surface the error to the creator.
- Never modify `.env` or secrets — only use what's already configured.

### Cloud-provider auth drift

If `preflight.*` flags a health-check that works against the provider but doesn't match
the shape in `seed/tech-stack.md` (common example: Supabase new-key projects require
`apikey` + `Authorization: Bearer`, not just `apikey`), follow
`SeedKit-v2/approaches/claude-code/preflight-pattern.md §Cloud-provider auth drift`:

1. **Fix `dev-runbook.md` immediately** — it's editable, outside `seed/`. Replace the stale
   curl with the working one. Add a parenthetical noting the discrepancy with the seed and
   which `seed/…` sections contain the stale version.
2. **Append an `OQ-PREFLIGHT-*` entry to `seed/open-questions.md`** — only file in `seed/`
   that's append-only during implementation. Document the discrepancy and the committed
   default the runbook now applies.
3. **Update the relevant Serena memory** — usually `suggested_commands` — so the next
   session starts from the corrected shape.
4. **Leave `seed/` files untouched** until the creator approves the seed edit.

This is the general seed-drift pattern: editable files absorb the fix now; the OQ is the
audit trail; the seed catches up on the next creator review.

## Workflow — phase-driven

### The four artifacts and what each owns

```
seed/roadmap.md               ← phases + exit criteria; master sequencing (read-only during impl.)
plans/NNNN-phase-slug.md      ← one plan per roadmap phase; atomic subtasks + per-subtask verification
SESSION_HANDOFF.md            ← living project state: what shipped, what's in progress, what's next
CHANGELOG.md                  ← immutable log populated from commits; SESSION_HANDOFF links, never duplicates
```

### Starting a new phase

1. Read `seed/roadmap.md` for the next phase: scope, user stories, out-of-scope, exit criteria.
2. Generate `plans/NNNN-phase-slug.md` (use `/ultraplan` if available, or draft manually). The plan must:
   - Cite the seed sections it satisfies (file + section number)
   - List user story IDs from `seed/user-stories.md` this phase covers
   - Name the non-negotiables from `seed/creator-vision.md` that apply
   - Enumerate atomic subtasks, each with a per-subtask verification command or assertion
   - Include the phase exit criteria from `seed/roadmap.md` verbatim in a `## Gate` section
3. Get plan approval before writing any code.

### Working within a phase

- **Tests are written inside the subtask, not at the end.** Implementing a subtask means writing
  the code AND its test together. The `Verify:` command runs that test to confirm the subtask
  is complete. The phase gate then runs the full suite over everything shipped in the phase.
- Run each subtask's `Verify:` before starting the next. Don't stack unverified work.
- If reality diverges from the plan (unexpected dep, wrong assumption): update the plan first, then continue.
- Do NOT create a new plan file for a subtask — subtasks live within the phase plan.

### Closing a phase

1. Run all exit criteria from `seed/roadmap.md` for this phase. Every command must pass.
2. Run the mandatory Verification ritual (all 9 steps below — type/lint → gate → memory hygiene).
3. Mark the phase ✅ in `seed/roadmap.md`.
4. Append to `CHANGELOG.md`: `## YYYY-MM-DD — Phase N: [name]` + one bullet per shipped story.
5. Update `SESSION_HANDOFF.md`: mark phase shipped in the table, point to the next phase.
   Cross-reference any `decision_*_YYYY_MM_DD` memories written this phase.
6. Single commit with all gates green (including Serena memory writes from step 9).
   Message format: `feat(phase-N): [scope description]`.

### When to use `/ultraplan`

Two moments only:
1. During seed authoring — to generate `seed/roadmap.md` if not yet written
2. At the start of a new roadmap phase — to generate the phase plan file

Do not use it for subtasks within a phase — those are bullets in the plan, not new plan files.

## Plan format — `plans/NNNN-phase-slug.md`

```markdown
# Phase N — [Name]

**Seed sections:** [list of seed file + section citations driving this plan]
**User stories:** [story IDs from user-stories.md]
**Non-negotiables:** [list from creator-vision.md that apply]

## Scope
[What this phase delivers. Out-of-scope explicit.]

## Subtasks

### 1. [Name]
[What to build]
**Verify:** [the specific command that proves this subtask is done — see testing stack below]

### 2. [Name]
...

## Gate
[Exact exit criteria from seed/roadmap.md for this phase, copied verbatim]

## Verification
[Filled in after implementation: code-review findings, code-simplifier deltas, test run results]
```

**What goes in `Verify:` per subtask** — pick the command appropriate for the project type,
sourced from `seed/tech-stack.md` testing stack:

| Project type | Example Verify: commands |
|---|---|
| TypeScript (any) | `tsc --noEmit` · `vitest run src/module.test.ts` |
| Python (any) | `pytest tests/unit/test_module.py -v` · `mypy src/module.py` |
| REST API | `curl -s localhost:PORT/endpoint \| jq` → expected JSON |
| CLI tool | `./bin/tool [args]` → expected stdout output |
| UI / web | `playwright test e2e/feature.spec.ts` · `vitest run src/component.test.tsx` |
| DB migration | `[migration tool] status` → migration applied; query returns expected row |
| Background job | trigger job → poll result endpoint or log → expected output |

The `Verify:` command for a subtask should be **runnable in isolation** and prove only that subtask's
work — typically the specific test file(s) written alongside the implementation code.

The `## Gate` proves the whole phase: it copies the exit criteria from `seed/roadmap.md` verbatim
and runs the full test suite + any health checks. A subtask's `Verify:` passing does not substitute
for the Gate — the Gate is the final authority before committing.

## Verification ritual (mandatory before closing a phase)

**The testing commands for this project come from `seed/tech-stack.md` "Testing stack" section.**
The ritual below is the order — the specific commands depend on the project type.

Every phase passes these checks before the gate is run:

1. **Type/lint check** — run the static analysis tool declared in the testing stack
   (e.g. `tsc --noEmit`, `mypy`, `cargo check`, `go vet`). Zero errors required.
2. **Unit test suite** — run the unit test runner for the modules changed in this phase.
   (e.g. `vitest run`, `pytest tests/unit/`, `cargo test --lib`, `go test ./...`)
3. **Integration tests** — if the phase touches external boundaries (DB, API, file I/O, queue):
   run the integration test suite declared in the testing stack.
   If no integration tests exist yet for this phase, note it explicitly and add them.
4. **E2E / acceptance tests** — only if the project has a navigable UI or testable CLI:
   - UI project: drive the relevant flows from `seed/user-stories.md` via playwright
   - CLI project: invoke the CLI with the expected inputs and assert the output
   - API project: run contract tests or curl assertions against the running server
   Record which flows/scenarios were exercised in the plan's `## Verification` section.
5. **`code-review` pass** — findings that reveal seed misreads are fixed;
   ambiguities go to `seed/open-questions.md`.
6. **`code-simplifier` pass** — strip speculative abstractions, unused scaffolding,
   premature generalizations. Does NOT override seed-required surfaces.
7. **Non-omission check** — verify no required behavior, state, flow, or non-negotiable was dropped.
8. **`seed/roadmap.md` gate** — run all exit criteria commands for this phase verbatim. All must pass.
9. **Memory hygiene** *(blocker before the phase commit)* — update Serena memories the phase
   affected, so the next session starts from current truth, not the state captured at last
   onboarding. Stale memories are worse than no memories.
   - **Minimum:** call `mcp__serena__write_memory` (or `edit_memory` for incremental) on
     `structure_and_layout` with the new paths/files that now exist vs the target shape.
   - **If new runtime scripts landed** (e.g. `pnpm`/`npm`/`make`/`cargo` targets) → also
     update `suggested_commands`.
   - **If the tech stack shifted** (version bump, new pinned dep, prohibition lifted by a
     seed edit) → also update `tech_stack`.
   - **If a convention was added or clarified** (new rule, new identifier grammar, new
     validation pattern) → also update `style_and_conventions`.
   - **If the phase closed user stories / flows / non-negotiables** → also update
     `project_overview` (status line + any scope delta).
   - Non-trivial mid-phase decisions that don't fit any of the six canonical memories go to
     a dedicated `decision_*_YYYY_MM_DD` memory — see §Mid-phase decision memories.

## Serena (semantic code navigation)

The Serena MCP is the **default** for code exploration and symbol-level edits. It is installed
**hybrid** — the MCP server from the Anthropic marketplace plugin (`plugin:serena:serena`)
and the CLI binaries (`serena` + `serena-hooks`) from a separate `uv tool install serena-agent`.
Both are required for the 4 hooks in `.claude/settings.json` to fire (activate / remind /
auto-approve / cleanup). See `SeedKit-v2/approaches/claude-code/serena-playbook.md` for the
full playbook — install paths, decision matrix, memories workflow, and per-language notes.

**Prefer Serena tools over `Read` / `Grep` / line-based `Edit` for any code-reading or
code-editing task.** The `PreToolUse → remind` hook enforces this at every tool call — don't
fight it. In this install the tools are namespaced `mcp__plugin_serena_serena__*`; the short
references below use the documentation shorthand `mcp__serena__*`.

### Hard rule

> If the file is **> ~150 lines** OR you **don't yet know which symbol you want**, start with
> `mcp__serena__get_symbols_overview`. Never `Read` the whole file as the first action.

### The short version

- Map a file: `mcp__serena__get_symbols_overview` (depth=1).
- Read one function/class: `mcp__serena__find_symbol name_path=... include_body=true`.
- Find callers before a refactor: `mcp__serena__find_referencing_symbols`.
- Edit a symbol body: `mcp__serena__replace_symbol_body` (not line-based `Edit`).
- Add a new method: `mcp__serena__insert_after_symbol` / `insert_before_symbol`.
- Rename everywhere: `mcp__serena__rename_symbol`.
- Recall prior findings: `mcp__serena__list_memories` + `mcp__serena__read_memory`.
- Persist a non-trivial finding: `mcp__serena__write_memory` (full overwrite) or
  `mcp__serena__edit_memory` (regex-replace for small incremental additions).
- Capture a non-obvious trade-off made mid-phase: `mcp__serena__write_memory` with name
  `decision_<slug>_YYYY_MM_DD` — see §Mid-phase decision memories.

Every phase plan file carries a "Serena-first" preamble at the top — it consults the playbook
automatically. If you're drafting a plan manually (without `/ultraplan`), include the preamble
exactly as documented in the playbook §How the phase plan and kickoff use this playbook.

### Two complementary tracking layers

- **`SESSION_HANDOFF.md`** (human-readable) — living project state: current phase + subtask,
  shipped phases table, active stack config, implementation decisions, coming next.
- **`.serena/memories/`** (agent-readable) — six canonical memories + `decision_*_YYYY_MM_DD`
  memories. Auto-hydrated by the `SessionStart → activate` hook.

They don't duplicate each other — `SESSION_HANDOFF.md` is what the person reads at session
start; Serena memories are what the agent loads to reconstruct context deterministically.
The Verification ritual keeps both in sync at phase close.

## Mid-phase decision memories

When a **non-trivial decision** surfaces mid-phase — one that a future session working in
the same area would otherwise have to re-derive — capture it as its own Serena memory **the
moment it's decided**, not at phase close. The plan is a forward contract; `SESSION_HANDOFF.md`
is living state; decision memories are operational context that survives across phases.

### What counts as "non-trivial"

- A choice between ≥2 valid approaches where the rejected path has non-obvious cost.
- A workaround for a library / platform quirk with a citation (framework constraint,
  database driver behavior, SDK edge case).
- A schema or data-model trade-off that's not captured verbatim in `seed/schema.md`
  (denormalization for a hot read path, computed column vs trigger, etc.).
- A convention extension introduced for a new surface (e.g. testid grammar for a new
  screen, new error-code prefix, new Zod-schema naming) so downstream work stays consistent.
- A seed-ambiguity that was resolved in code before `seed/open-questions.md` was updated
  (should also push the resolution into the OQ as the append-only log, but the memory
  captures the "why we went this way" in one place).

### What does NOT go here

- Anything already in `seed/` — the seed wins; don't duplicate.
- Anything a fresh reader could derive from a single `get_symbols_overview` + `find_symbol`
  call. Code is self-documenting for local decisions.
- Ephemeral task progress — that belongs in `SESSION_HANDOFF.md`, not in a Serena memory.

### Naming convention

`decision_<slug>_YYYY_MM_DD` — slug is 1-4 kebab-words summarizing the area. The date lets
a future session see *when* the decision was made (and therefore what state the codebase
was in at the time). Examples:

- `decision_host_middleware_2026_04_23`
- `decision_write_through_cache_2026_05_02`
- `decision_event_naming_grammar_2026_05_18`

### Shape

```markdown
# <one-line title — what was decided>

**Decided:** YYYY-MM-DD during Phase N subtask X.Y
**Seed anchors:** [citations from seed/]
**Related OQ:** [if any — include the committed default]

## Options considered
1. <Option A> — pros / cons
2. <Option B> — pros / cons (this is what shipped)

## Why the chosen path
<2–4 sentences. The *why*, not the *what*.>

## What a future session needs to know
- <Invariant or trap that could regress the decision>
- <File(s) or symbol(s) that embody it>
- <The symptom if someone violates it>
```

### Write, then move on

```text
mcp__serena__write_memory
  memory_name: decision_<slug>_YYYY_MM_DD
  content: <the shape above>
```

No need to wait for phase close — capture the decision **in the same turn it's made**. At
phase close, the memory-hygiene step (Verification ritual step 9) cross-references any
decision memories written this phase in the phase's entry in `SESSION_HANDOFF.md`.

## Guardrails

- **Non-invention.** If something is not in the seed, do not invent it. Flag it in `seed/open-questions.md`.
- **Non-omission.** Required screens, states, flows, and non-negotiables are not droppable.
- **Non-ambiguity.** A phrase that can be interpreted two ways is a defect. Surface it; do not resolve in code.
- **No micro-plans.** Do not create a new plan file for a subtask within the current phase.
- **No WIP commits.** Single commit per phase with all gates green.
- **SESSION_HANDOFF.md discipline.** It tracks state, not history. If it starts accumulating
  narrative prose per cycle or duplicating seed content, it has drifted — restructure it.
  Dense paragraphs → CHANGELOG.md. Architecture → seed/. Rules → this file.
- **Seed is read-only during implementation.** Update only `seed/open-questions.md` (append-only).
  Any other seed change requires creator approval and a version note in `seed/creator-vision.md`.
- **Don't fight the Serena hooks.** The `PreToolUse → remind` hook nudges you toward symbolic
  tools; the `auto-approve` hooks keep `replace_symbol_body` / `rename_symbol` flowing without
  prompts. If a hook blocks you unexpectedly, surface it — don't disable it.
- **Memory hygiene is a commit blocker.** Stale Serena memories are worse than no memories.
  Run Verification ritual step 9 before every phase commit.

## Escalation rules

Stop and surface to the creator if:
- A non-negotiable from `creator-vision.md` would be violated
- A seed file contradicts another and the conflict is material
- An open question in `seed/open-questions.md` blocks implementation of the current phase
- A library API does not match what context7 returns and the seed does not specify an alternative
- Three consecutive failed attempts to start or restart a dev service
