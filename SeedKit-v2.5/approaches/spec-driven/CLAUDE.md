## Source of truth

The `seed/` folder is the sole authoritative source for what to build.
`constitution.md` at the project root is a condensed extract of `seed/creator-vision.md`
non-negotiables — it is the quick-check artifact for spec and plan phases.
On any conflict: `seed/creator-vision.md` wins over `constitution.md`.

Document precedence within `seed/` (higher wins on conflict):

1. `seed/creator-vision.md` — absolute authority; creator intent and non-negotiables
2. `seed/user-stories.md` — user-centered behaviors; primary source for spec.md user stories
3. `seed/product.md` *(if present)* — scope and out-of-scope
4. `seed/domain.md` *(if present)* — domain invariants and canonical terminology
5. `seed/schema.md` — data model and isolation rules; primary source for data-model.md
6. `seed/ux.md` *(if present)* — screens, flows, required states; source for contracts/ shapes
7. `seed/design.md` *(if present)* — visual brief; pointer to design-system/
8. `seed/tech-stack.md` — stack, prohibitions, testing; populates plan.md Technical Context
9. `seed/roadmap.md` — phase sequence and done-criteria; determines the feature spec order
10. `seed/open-questions.md` — append-only; ambiguities go here, not to code

## Workflow — feature spec-driven

### The per-feature artifact set

```
specs/[###-feature-name]/
├── spec.md          ← Phase 0: requirements (user stories + acceptance scenarios)
├── plan.md          ← Phase 1: technical plan (architecture, deps, file structure)
├── data-model.md    ← Phase 1: schema changes for this feature (if applicable)
├── contracts/       ← Phase 1: API endpoint shapes (if applicable)
├── research.md      ← Phase 1: library/API research (context7 output)
└── tasks.md         ← Phase 2: executable tasks; generated LAST from plan + data-model + contracts
```

Tasks are organized by user story — each story implementable and testable independently.
`[P]` on a task = can run in parallel with other `[P]` tasks on different files.

### Phase 0 — Specify

1. Identify the next feature from `seed/roadmap.md`.
2. Create a feature branch: `git checkout -b [###-feature-name]`
3. Create `specs/[###-feature]/spec.md`:
   - User stories with priorities (P1/P2/P3) — sourced from `seed/user-stories.md`
   - Acceptance scenarios (Given/When/Then) — sourced from `seed/ux.md` flows and states
   - Non-negotiables that apply — from `seed/creator-vision.md`
   - Explicit out-of-scope for this feature
   - Seed citations (which file + section each story/scenario comes from)
4. Get creator approval. Do NOT proceed to Phase 1 without it.

### Phase 1 — Plan

After spec.md is approved:

1. Create `specs/[###-feature]/plan.md`:
   - Technical approach and rationale
   - Dependencies (query context7 for any library API before specifying)
   - Concrete file structure (source + test paths)
   - Constitution check: verify all non-negotiables from `constitution.md` are respected
2. Create `specs/[###-feature]/data-model.md` if the feature touches the schema
3. Create `specs/[###-feature]/contracts/` if the feature adds or changes API endpoints
4. Create `specs/[###-feature]/research.md` with context7 findings for any library decisions
5. Get creator approval. Do NOT proceed to Phase 2 without it.

### Phase 2 — Tasks

After plan.md is approved:

1. Generate `specs/[###-feature]/tasks.md` from plan.md + data-model.md + contracts/
   - Organized by user story (US1, US2, US3...)
   - Each task: `[ID] [P?] [USN] Description with exact file path`
   - [P] tasks can run in parallel — mark explicitly
2. Implement task by task. Check `[ ]` → `[x]` as each is done.
3. After each task: run the relevant verification (unit test, typecheck, or playwright assertion)
4. Do NOT start the next task if the current one's verification fails.

### Closing a feature

1. All tasks in tasks.md are `[x]`.
2. Run all acceptance scenarios from spec.md — all must pass.
3. Run mandatory passes: code-review → code-simplifier → playwright (for UI features).
4. Append to `CHANGELOG.md`: `## YYYY-MM-DD — [Feature name]` + one bullet per user story shipped.
5. Update `SPEC-STATUS.md` — mark feature done, point to next feature.
6. Single commit per feature with all gates green. Merge PR. Message: `feat([###-slug]): [description]`

## Verification ritual (mandatory before closing a feature)

1. **All acceptance scenarios from spec.md pass** — run playwright or equivalent for each
2. **code-review pass** — findings that reveal seed misreads are fixed;
   ambiguities go to `seed/open-questions.md`
3. **code-simplifier pass** — strip speculative abstractions and unused scaffolding
4. **Non-omission check** — no required screen, state, flow, or non-negotiable was dropped
5. **Constitution check** — re-verify `constitution.md` principles are respected in the shipped code

## Plugins and tools

### Universal

| Tool | When |
|---|---|
| `context7` | Any library API decision — before writing spec scenarios or plan contracts |
| `playwright` | Any UI feature — acceptance scenarios must be playwright-runnable |
| `serena` | Symbol-level navigation in large codebases |

| Plugin | When |
|---|---|

| `code-simplifier` | After tasks complete, before final acceptance run |
| `frontend-design` | Any UI feature — cross-reference against design-system/ |
| `feature-dev` | Complex multi-file features — primary driver for plan.md generation |

| Command | What it does |
|---|---|
| `/speckit.specify` | Creates specs/[###-feature]/spec.md from a description |
| `/speckit.plan` | Creates plan.md + data-model.md + contracts/ + research.md from spec.md |
| `/speckit.tasks` | Generates tasks.md from plan.md + data-model.md + contracts/ |

If spec-kit is not installed, create these files manually following the formats above.

## Guardrails

- **Non-invention.** Spec scenarios and plan decisions must cite a seed source. If not in seed, flag in `seed/open-questions.md`.
- **Non-omission.** Every P1 user story in the spec must ship in the feature. P2/P3 can defer with explicit out-of-scope note.
- **No code before approval.** spec.md needs approval before plan.md. plan.md needs approval before tasks.md and implementation.
- **Tasks from plan, not from memory.** Generate tasks.md FROM plan.md + data-model + contracts. Do not write tasks from scratch.
- **Seed is read-only.** Update only `seed/open-questions.md` (append-only) during implementation.

## Escalation rules

Stop and surface to creator if:
- A non-negotiable from `seed/creator-vision.md` would be violated by the spec or plan
- A seed file contradicts the spec you are writing and the conflict is material
- A context7 result shows the library API is incompatible with the spec's acceptance scenarios
- Three consecutive dev service restart failures
