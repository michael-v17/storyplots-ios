# Generation Guide: seed/tech-stack.md

## 1. When to include

**Always.** It is a conditional enricher in the spark files, but in practice it must always exist — the roadmap needs the testing stack for its executable exit criteria. It can be generated in a session after schema and ux, but before the roadmap.

---

## 2. Prerequisites

These must exist before opening this session:

- [ ] `seed/creator-vision.md` — explicit stack preferences and prohibitions
- [ ] `seed/schema.md` ✓ approved — determines DB, auth, and RLS needs
- [ ] `seed/references/[AppName]/07-technical-notes.md` *(or equivalent)* — stack observed in the reference

`seed/ux.md` helps (determines whether there is SSR vs SPA, server components, etc.) but is not a blocking prerequisite.

---

## 3. Session opening prompt

```
We are going to generate seed/tech-stack.md. This file defines the chosen stack with
pinned versions, explicit prohibitions, the testing stack, and layer responsibilities.

Read the following files in this order before starting:
1. seed/creator-vision.md                              — stack preferences + prohibitions
2. seed/schema.md                                      — DB, auth, RLS needs
3. seed/references/[AppName]/07-technical-notes.md     — stack observed in the reference

Produce seed/tech-stack.md with this structure:
- Layers table: language, runtime, framework, DB, auth, testing, tooling
  with choice, exact pinned version, and rationale
- Section "What we are NOT using" — explicit prohibitions with rationale
- Testing stack: unit runner + e2e tool + what each one covers
- Layer responsibilities: what frontend does, what backend does, what agents do
- Dev services table: for each service → name, EXACT start command, port,
  health check command, log location
  (this table is the source the kickoff uses to generate dev-runbook.md)
- Also produce a `.env.example` file at the project root with all required environment
  variable names and placeholder values (no real secrets). One line per variable,
  grouped by service. This file is generated here because the tech stack determines
  which variables are needed.

CRITICAL RULES:
- Exact versions ALWAYS (not ^1.0 or ~1.0 — only exact 1.0.0)
- Every prohibition MUST have a rationale ("no X because Y")
- Do NOT include run commands (those go in dev-runbook.md, generated by the kickoff)
- Do NOT include domain concepts, schema, or screen specs
- If creator-vision says "no Docker" → it appears in "What we are NOT using" verbatim
- If there is ambiguity between what is observed in the reference and what creator-vision requests →
  creator-vision wins; document the divergence with "(creator-vision §X overrides observed)"
```

## 4. Extraction map

### From `seed/creator-vision.md`
- All explicit technology mentions: "I want FastAPI", "use React", "no Docker for now"
- Explicit prohibitions or declared stack anti-patterns
- Infrastructure requirements: BYOK, streaming (SSE vs WebSocket), cloud provider
- Simplicity vs power preferences: "keep it simple", "no ORM", etc.
- Any mention of specific versions or ecosystem preferences

### From `seed/schema.md`
- Type of database required (relational, document, etc.)
- RLS/auth policy needs (determines whether it is Supabase, PostgreSQL + custom auth, etc.)
- Volume or access patterns that influence the DB choice
- Entities that require streaming or real-time (determines whether Supabase Realtime, SSE, WebSocket is needed)

### From `seed/references/[AppName]/07-technical-notes.md`
- Observed stack (framework, DB, auth provider)
- Observed technical patterns (SSE, REST, GraphQL, WebSocket)
- Observed performance or scale constraints
- Observed cloud provider or infrastructure

## 5. Required output structure

### 5.1 Layers table

| Layer | Choice | Version (pinned) | Rationale |
|---|---|---|---|
| Language | TypeScript | 5.4.5 | Type safety; creator-vision §3 |
| Runtime | Node.js | 20.12.2 LTS | LTS stability |
| Frontend framework | React | 18.3.1 | Reference app observed; creator-vision §3 |
| Backend framework | Hono | 4.3.11 | Lightweight; creator-vision: "keep it simple" |
| Database | Supabase (PostgreSQL) | 2.43.0 (client) | RLS + auth; schema.md isolation rules |
| Auth | Supabase Auth | built-in | Same provider as DB; BYOK requirement in schema |
| Unit testing | Vitest | 1.6.0 | Fast, TS-native |
| E2E testing | Playwright | 1.44.0 | creator-vision: Playwright preferred |
| Package manager | pnpm | 9.1.0 | Workspace support; faster than npm |
| ... | ... | ... | ... |

**Table notes**:
- The "Version" column must be an exact number, never a range
- The "Rationale" column cites creator-vision or schema when it is a non-obvious decision
- Include the client/SDK version for external services (Supabase client, Anthropic SDK, etc.)

### 5.2 "What we are NOT using"

**This section is the most valuable for preventing scope creep.** Explicit prohibitions:

```markdown
## What we are NOT using

| Not using | Rationale |
|---|---|
| Docker / containers | creator-vision: "keep dev simple, no Docker for now" |
| ORM (Prisma, Drizzle, TypeORM) | Direct SQL via Supabase client; creator-vision: "no ORM" |
| Redux / Zustand | React Query + local state; creator-vision: keep it simple |
| WebSockets | SSE for streaming; simpler for this use case |
| React Native | Web-first; creator-vision §2 platform scope |
| GraphQL | REST + RPC; no schema complexity that warrants it |
| ... | ... |
```

Include in this section:
- Any technology someone might "reasonably" suggest but the creator rejected
- Alternative versions of something included (e.g.: if using React 18, not React 17)
- Excluded patterns (no ORM, no SSR in this project, no feature flags)

### 5.3 Testing stack

```markdown
## Testing stack

| Tool | Version | Covers | Does NOT cover |
|---|---|---|---|
| Vitest | 1.6.0 | Unit tests: services, utilities, pure functions | UI interactions, DB integration |
| Playwright | 1.44.0 | E2E flows: F1..F7 critical flows, smoke tests | Unit logic, pure functions |

**Unit test scope:** any function/service that has business logic, data transformation,
or algorithm. Minimum: one test per public service method.

**E2E test scope:** each critical flow F1..FN from seed/user-stories.md §6.
Tests run against a local dev server with test fixtures.

**What we don't test:** third-party integrations (Anthropic API, Supabase internals).
We mock at the service boundary.
```

### 5.4 Layer responsibilities

```markdown
## Layer responsibilities

**Frontend (React):**
- Renders UI from server data
- Manages local UI state (open/closed, form state)
- Calls backend API via fetch/React Query
- Does NOT: run agents, hold API keys in plaintext, implement business logic

**Backend (Hono API):**
- Validates requests, enforces auth
- Orchestrates agents via LangGraph
- Reads/writes Supabase (source of truth)
- Does NOT: hold conversation state in memory between requests, run UI logic

**Agents (LangGraph):**
- Execute the AI flows (conversation, grammar, memory)
- Receive isolated context per Conversation
- Does NOT: access other Conversations' data, write directly to DB (via service layer only)
```

Layer responsibilities in tech-stack.md are at the level of "what each layer does". For multi-service projects with complex integrations, responsibilities are expanded in `architecture.md`.

### 5.5 Dev services table

**This section is the source of truth for `dev-runbook.md`.** The kickoff reads it to generate the runbook with exact commands. If this table is incomplete or generic, the runbook will be useless and the AI will not be able to manage the development environment autonomously.

For each service in the project:

| Service | Start command (exact) | Port | Health check command | Logs location |
|---|---|---|---|---|
| Frontend | `pnpm --filter frontend dev` | 5173 | `curl -s localhost:5173 \| head -1` → `<!DOCTYPE html>` | process stdout |
| Backend | `pnpm --filter api dev` | 3000 | `curl -s localhost:3000/health` → `{"status":"ok"}` | process stdout |
| Supabase local | `supabase start` | 54321 | `supabase status` → `API URL: http://...` | `supabase logs` |

**Rules for this table:**
- "Start command" must be the literal and complete command — the AI copies it and executes it without modification
- "Health check command" must verify that the service is WORKING, not just running (a process can be alive but in an error state)
- If the project does not use local DB services (uses Supabase cloud) → document that the DB is cloud and what to verify (e.g.: `supabase db ping`)
- Include the startup order if there are dependencies (e.g.: "DB must be ready before the backend")

**Note:** run commands do NOT go in the rest of the document — ONLY in this §Dev services section.

## 6. Quality gates before approving

- [ ] All technology mentions in `creator-vision.md` are reflected (search for named tools)
- [ ] Every prohibition in "What we are NOT using" has an explicit rationale
- [ ] All versions are exact (search for `^` and `~` — they must not appear in the table)
- [ ] Testing stack declares which runner for unit and which tool for e2e
- [ ] There are no run commands in the document (`npm run dev`, `pnpm start` — those go in dev-runbook.md)
- [ ] Layer responsibilities include "Does NOT" for each layer
- [ ] The testing stack is consistent with what the roadmap will need for its exit criteria
- [ ] The §Dev services section exists with one row per service in the project
- [ ] Each Dev services row has an exact command, port, and verifiable health check command

---

## 7. Common failure modes

**1. Versions with ranges (`^` or `~`)**
The AI writes `"react": "^18.3.0"`. Detect: search for `^` or `~` in the document. Solution: replace with exact version. The SEED-GUIDE.md Rule 17 says: pinned versions, no ranges.

**2. Omitting "What we are NOT using"**
The AI lists the stack but does not state what is explicitly out. Consequence: in future sessions, the AI suggests using Prisma or Docker because "it's not prohibited". Detect: verify that the section exists. Solution: add it before approving, explicitly consulting creator-vision for prohibitions.

**3. Including run commands**
The AI writes `pnpm dev` or `npm run build` in the tooling section. Those commands go in `dev-runbook.md` (generated by the kickoff). Detect: search for commands like `npm`, `pnpm`, `yarn` followed by `run`, `start`, `build`. Solution: remove them and note that they go in dev-runbook.

**4. Layer responsibilities without explicit "Does NOT"**
The AI describes what each layer DOES but does not say what it does NOT DO. Without that, the implementer does not know whether they can put business logic in the frontend or whether agents can write directly to the DB. Detect: verify that each layer has a "Does NOT" section. Solution: add it before approving.

**5. Testing stack without declared scope**
The AI lists the tools but does not say what each one covers. Consequence: the roadmap cannot generate executable exit criteria without knowing whether to use vitest or playwright for each assertion. Detect: verify that each tool has "Covers" and "Does NOT cover". Solution: complete it before approving.

**6. Dev services table absent or with generic commands**
The AI omits §Dev services, or writes commands like `npm start` instead of the project's real command. Direct consequence: when the kickoff generates `dev-runbook.md`, the commands are incorrect and the AI cannot start the development environment — the creator has to do it manually before each session. Detect: verify that §Dev services exists and that each command can be copied and executed without modification. Solution: complete it before approving; if the exact commands are not yet known, leave placeholders `[TODO: verify exact command]` instead of inventing.
