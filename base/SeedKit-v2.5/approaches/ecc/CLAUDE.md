## Starting a new session — READ THIS FIRST

1. **Run preflight first.** `./preflight.ps1` (Windows) or `./preflight.sh` (macOS/Linux/WSL)
   from the repo root. It verifies pinned tools are on PATH, `.env.local` parity with
   `.env.example`, opaque-key shape, and live reachability of external services — all
   without printing any secret values. Expected: `HTTP 200` on the reachability line. If
   it fails, fix the failure before coding. See `SeedKit-v2/approaches/ecc/preflight-pattern.md`
   for the pattern's rationale and the drift-handling workflow.
2. Check `roadmap.md` at the project root → find the row marked `in-progress` (or the `Next` pointer).
3. Check ECC session memory + `.serena/memories/` → six canonical memories summarise
   project / tech-stack / commands / style / task-completion / structure. Any additional
   `decision_*_YYYY_MM_DD` memories describe non-obvious trade-offs you may need before
   editing specific areas (see §Mid-phase decision memories).
4. Open the current phase's plan file (path is in the `roadmap.md` row).
5. Start dev services from `dev-runbook.md` → verify all healthy before coding.

If ECC memory is empty or stale: `roadmap.md` is the source of truth. The last `done` row is what
was last shipped; the row marked `in-progress` (if any) is where to resume; otherwise the `Next`
pointer names the phase to plan.

## Source of truth

The `seed/` folder is the sole authoritative source for **what to build**.
The ECC harness (`.claude/`) configures **how** Claude works — it does not define what to build.
If an ECC agent suggestion conflicts with the seed, the seed wins.

Document precedence within `seed/` (higher wins on conflict):

1. `seed/creator-vision.md` — absolute authority
2. `seed/user-stories.md` — what users must do
3. `seed/product.md` *(if present)* — scope and out-of-scope
4. `seed/domain.md` *(if present)* — domain invariants
5. `seed/schema.md` — data model and isolation rules
6. `seed/ux.md` *(if present)* — screens, flows, required states
7. `seed/design.md` *(if present)* — visual brief; pointer to design-system/
8. `seed/tech-stack.md` — stack, prohibitions, testing stack
9. `seed/roadmap.md` — phase sequence with executable exit criteria
10. `seed/architecture.md` *(if present)* — subsystem responsibilities
11. `seed/open-questions.md` — append-only; ambiguities go here
12. `seed/references/[AppName]/` *(if present)* — interpretive only; the observed-app evidence base
    from which Layer 2 seed files were generated. Consult when seed is silent on a behavior detail;
    never authoritative over seed files above.

`roadmap.md` at the project root is a **projection** of `seed/roadmap.md` plus live status. It is the
only non-seed file that is human-editable to reorder or insert phases.

## Dev environment

Read `dev-runbook.md` at session start. Confirm all services are running before coding.

- All services: `Bash(run_in_background=true)`. Never foreground.
- Logs: `BashOutput(id)`. Kill: `KillShell(id)`.
- 3 failed restarts → escalate to creator.
- Never modify `.env` or secrets.

### Cloud-provider auth drift

If `preflight.*` flags a health-check that works against the provider but doesn't
match the shape in `seed/tech-stack.md` (common example: Supabase new-key projects
require `apikey` + `Authorization: Bearer`, not just `apikey`), follow
`SeedKit-v2/approaches/ecc/preflight-pattern.md §Cloud-provider auth drift`:

1. Fix `dev-runbook.md` immediately (it's editable, outside `seed/`).
2. Append an `OQ-PREFLIGHT-*` entry to `seed/open-questions.md` documenting the
   discrepancy and the committed default.
3. Update the relevant Serena memory (`suggested_commands` usually) so the next
   session starts from the corrected shape.
4. Leave `seed/` files untouched until the creator approves the seed edit.

## PRP workflow loop (the canonical cycle)

Every non-trivial phase runs through the same five-command cycle. Deviations need a reason.

```
roadmap.md (find Next pending phase)
   │
   ▼
/prp-plan <phase brief>
   │   writes .claude/PRPs/plans/NNNN-phase-N-<slug>.plan.md
   │   flips roadmap.md row to in-progress
   ▼
(human review of the plan — edit if needed)
   │
   ▼
/prp-implement .claude/PRPs/plans/NNNN-phase-N-<slug>.plan.md
   │   writes code; 5-level validation after every task
   │   writes .claude/PRPs/reports/NNNN-phase-N-<slug>-report.md
   ▼
/code-review  +  /quality-gate
   │
   ▼
/prp-commit "phase N — <scope>"
   │
   ▼
/prp-pr
   │
   ▼
roadmap.md row → done; Next pointer advances
```

Single-commit-per-phase discipline is preserved by `/prp-commit` — no WIP commits inside a phase.
If a phase cannot fit in one commit, split it into two phases **in `roadmap.md`** before continuing.

## Roadmap.md contract

`roadmap.md` at the project root tracks phase status. Columns:

| # | Phase | Status | Plan | Report | Notes |

- **Status values:** `pending`, `in-progress`, `done`, `blocked`.
- **Who edits it:** `/prp-plan` writes the plan path and flips `pending → in-progress`.
  `/prp-implement` writes the report path and flips `in-progress → done`, then advances the `Next` pointer.
  A human may insert rows (new phases), reorder rows (priority change), or flip a row to `blocked`
  with a Note. Bulk rewrites are forbidden — phase history is audit trail.
- **The Next pointer** lives at the bottom of the file and names the next `pending` phase to plan.
  `/prp-plan` reads it to know where to start.

## Serena (semantic code navigation)

The Serena MCP is registered and indexed for this project (`.serena/project.yml` exists,
`.serena/memories/` is populated). Serena is installed **hybrid** — the MCP server comes
from the Anthropic marketplace plugin (`plugin:serena:serena`), and the CLI binaries
(`serena` + `serena-hooks`) from a separate `uv tool install serena-agent`. Both are
required for the 4 hooks in `.claude/settings.json` to fire (activate / remind /
auto-approve / cleanup). See `SeedKit-v2/approaches/ecc/serena-playbook.md` §One-time-setup
for the trade-off and alternative paths; see `dev-runbook.md` §One-time-setup for the
per-machine install commands.

**Prefer Serena tools over `Read` / `Grep` / line-based `Edit` for any code-reading or
code-editing task.** The `PreToolUse → remind` hook enforces this at every tool call —
don't fight it.

In this install the tools are namespaced `mcp__plugin_serena_serena__*`; the short
references below use the documentation shorthand `mcp__serena__*`.

### Hard rule

> If the file is **> ~150 lines** OR you **don't yet know which symbol you want**, start with
> `mcp__serena__get_symbols_overview`. Never `Read` the whole file as the first action.

### The short version

- Map a file: `mcp__serena__get_symbols_overview` (depth=1).
- Read one function/class: `mcp__serena__find_symbol name_path=... include_body=true`.
- Find callers before a refactor: `mcp__serena__find_referencing_symbols`.
- Edit a symbol body: `mcp__serena__replace_symbol_body` (not line-based `Edit`).
- Add a new method: `mcp__serena__insert_after_symbol` / `insert_before_symbol`.
- Rename everywhere: `mcp__serena__rename_symbol`.
- Recall prior findings: `mcp__serena__list_memories` + `mcp__serena__read_memory`.
- Persist a non-trivial finding: `mcp__serena__write_memory` (full overwrite) or
  `mcp__serena__edit_memory` (regex-replace for small incremental additions).
- Capture a non-obvious trade-off made mid-phase: `mcp__serena__write_memory` with name
  `decision_<slug>_YYYY_MM_DD` — see §Mid-phase decision memories.

`/prp-plan` and `/prp-implement` have a "Serena-first" preamble injected at the top — they
consult this playbook automatically. `/plan` does the same if it exists.

## Playwright capture (background)

**Only if this project reverse-engineers an external web app.** If not, ignore this section.

See `SeedKit-v2/approaches/ecc/playwright-capture-pattern.md` for the full pattern. Short rules:

- Never block on user input inside a capture script.
- Always launch spiders via `Bash(run_in_background=true)`; the conversation continues in parallel.
- Always use saved `playwright/.auth/state.json` (produced once via CDP handoff or cookie import;
  never type credentials in code).
- Always emit the 9-file packet per screen: `summary.md, aria.yaml, dom.html, screenshot-desktop.png,
  screenshot-mobile.png, requests.json, response-snippets.json, console.log, states-observed.md`.
- Always record `trace.zip` per journey; commit the derived `flow.md / state-matrix.md / requests.json`,
  gitignore the raw trace.
- If auth dies mid-run, re-run the handoff. Do NOT recover programmatically.
- Cookie **names** are documentable; cookie **values** are secrets. Never paste values.

## Workflow — phase-driven

ECC augments the tooling; it does not change the phase-driven discipline.

```
seed/roadmap.md                            ← phase sequence + exit criteria (read-only during impl.)
roadmap.md (project root)                  ← status projection; edited only by PRP commands + human inserts
.claude/PRPs/plans/NNNN-phase-*.plan.md    ← one plan per phase; atomic subtasks + per-subtask Verify
.claude/PRPs/reports/NNNN-phase-*-report.md ← implementation report written by /prp-implement
ECC memory                                 ← session continuity (replaces SESSION_HANDOFF.md)
CHANGELOG.md                               ← immutable shipped log (append on phase close)
```

### Starting a new phase

1. Check `roadmap.md`'s `Next` pointer.
2. Read `seed/roadmap.md` for the phase scope, stories, and exit criteria.
3. Run `/prp-plan <phase brief>`. The plan is written to
   `.claude/PRPs/plans/NNNN-phase-N-<slug>.plan.md` and `roadmap.md` flips to `in-progress`.
   The plan must:
   - Cite seed sections satisfied (file + section)
   - List user story IDs from `seed/user-stories.md`
   - Name non-negotiables from `seed/creator-vision.md` that apply
   - Enumerate atomic subtasks each with a `Verify:` command
   - Include phase exit criteria from `seed/roadmap.md` verbatim in `## Gate`
4. Get plan approval before writing code.

### Working within a phase

- **Tests are written inside the subtask**, not at the end — code + test together.
  The `Verify:` command runs that specific test. The `## Gate` runs the full suite.
- Run `Verify:` before starting the next subtask. Don't stack unverified work.
- ECC hooks run automatically (pre/post tool). Don't bypass them.
  If a hook blocks you unexpectedly, surface it — don't use `--no-verify`.

### Closing a phase

1. `/code-review` — findings addressed or escalated.
2. `/quality-gate` — all exit criteria green.
3. Append `CHANGELOG.md`: `## YYYY-MM-DD — Phase N: [name]` + one bullet per story.
4. **Memory hygiene** — run the §Verification ritual "Memory hygiene" step (update affected
   Serena memories: `structure_and_layout` at minimum; `suggested_commands` / `tech_stack` /
   `style_and_conventions` / `project_overview` / decision memories as applicable). This is a
   blocker for `/prp-commit` — stale memories are worse than no memories.
5. `/prp-commit "feat(phase-N): <scope>"` — single commit per phase. Memory writes from step 4 are part of this commit.
6. `/prp-pr` — opens the PR with plan + report links.
7. `/prp-implement`'s final step flips `roadmap.md` row to `done` and advances the `Next` pointer.

## Verification ritual (mandatory before closing a phase)

Testing commands come from `seed/tech-stack.md` "Testing stack" section.

1. **Type/lint check** — static analysis tool from the testing stack. Zero errors.
2. **Unit tests** — unit runner for modules changed in this phase.
3. **Integration tests** — if phase touches external boundaries (DB, API, file I/O).
4. **E2E / acceptance** — if project has UI: `/e2e` for relevant flows from `seed/user-stories.md`.
   If CLI: invoke with expected inputs, assert output.
5. **`/code-review` pass** — seed misreads are fixed; ambiguities → `seed/open-questions.md`.
6. **Simplifier pass** — strip speculative abstractions.
7. **Non-omission check** — no required behavior, state, or non-negotiable was dropped.
8. **Gate** — all `seed/roadmap.md` exit criteria for this phase. All must pass.
9. **Memory hygiene** *(before `/prp-commit`)* — update Serena memories the phase affected,
   so the next session starts from current truth, not the state captured at last onboarding.
   - **Minimum:** call `mcp__plugin_serena_serena__write_memory` (or `edit_memory` for
     incremental) on `structure_and_layout` with the new paths/files that now exist vs the
     target shape (new top-level folders, new migrations, new message/locale files, new
     modules — whatever the phase added).
   - **If new runtime scripts landed** (e.g. `pnpm`/`npm`/`make`/`cargo` targets) → also
     update `suggested_commands`.
   - **If the tech stack shifted** (version bump, new pinned dep, a prohibition lifted by a
     seed edit) → also update `tech_stack`.
   - **If a convention was added or clarified** (new rule file, new testid/identifier
     grammar, new validation pattern) → also update `style_and_conventions`.
   - **If the phase closed user stories / flows / non-negotiables** → also update
     `project_overview` (status line + any scope delta).
   - Use `write_memory` to overwrite in full; use `edit_memory` only for small incremental
     additions that are easier to express as a regex replace.
   - Non-trivial mid-phase decisions that don't fit any of the six canonical memories go to a
     dedicated decision memory — see `§Mid-phase decision memories` below.
10. **Tick `roadmap.md`** — `/prp-implement`'s final step flips the row to `done`.

## Mid-phase decision memories

When a **non-trivial decision** surfaces mid-phase — one that a future session working in
the same area would otherwise have to re-derive — capture it as its own Serena memory **the
moment it's decided**, not at phase close. The plan and the report are audit trails;
decision memories are operational context.

### What counts as "non-trivial"

- A choice between ≥2 valid approaches where the rejected path has non-obvious cost
  (e.g. "host-aware middleware over pure route groups because the subdomain cutover at
  launch must work without a router rewrite").
- A workaround for a library / platform quirk with a citation (framework constraint,
  database driver behavior, SDK cookie-handling edge case).
- A schema or data-model trade-off that's not captured verbatim in `seed/schema.md`
  (denormalization for a hot read path, computed column vs trigger, etc.).
- A convention extension introduced for a new surface (e.g. testid grammar for a new
  screen, new error-code prefix, new Zod-schema naming) so downstream work stays consistent.
- A seed-ambiguity that was resolved in code before `seed/open-questions.md` was updated
  (should also push the resolution into the OQ as the append-only log, but the memory
  captures the "why we went this way" in one place).

### What does NOT go here

- Anything already in `seed/` — the seed wins; don't duplicate.
- Anything a fresh reader could derive from a single `get_symbols_overview` + `find_symbol`
  call. Code is self-documenting for local decisions.
- Ephemeral task progress — that belongs in the phase report or ECC session memory, which
  have their own decay model.

### Naming convention

`decision_<slug>_YYYY_MM_DD` — slug is 1-4 kebab-words summarizing the area. The date
lets a future session see *when* the decision was made (and therefore what state the
codebase was in at the time). Examples:

- `decision_host_middleware_2026_04_23`
- `decision_write_through_cache_2026_05_02`
- `decision_event_naming_grammar_2026_05_18`

### Shape

```markdown
# <one-line title — what was decided>

**Decided:** YYYY-MM-DD during Phase N subtask X.Y
**Seed anchors:** [citations from seed/]
**Related OQ:** [if any — include the committed default]

## Options considered
1. <Option A> — pros / cons
2. <Option B> — pros / cons (this is what shipped)

## Why the chosen path
<2–4 sentences. The *why*, not the *what*.>

## What a future session needs to know
- <Invariant or trap that could regress the decision>
- <File(s) or symbol(s) that embody it>
- <The symptom if someone violates it>
```

### Write, then move on

```text
mcp__plugin_serena_serena__write_memory
  memory_name: decision_<slug>_YYYY_MM_DD
  content: <the shape above>
```

No need to wait for phase close — capture the decision **in the same turn it's made**.
At phase close, the memory-hygiene step cross-references any decision memories written
this phase in the phase report.

- Current phase and subtask (from the active plan file)
- Last commit SHA
- Any blockers
- Exact next action

## Guardrails

- **Non-invention.** Not in seed → flag in `seed/open-questions.md`, don't invent.
- **Non-omission.** Required behaviors, flows, non-negotiables are not droppable.
- **Seed over harness.** ECC agent suggestions that contradict the seed are wrong.
- **No WIP commits.** Single commit per phase with all gates green.
- **Seed is read-only.** Only `seed/open-questions.md` is append-only during implementation.
  Any other seed change requires creator approval.
- **`roadmap.md` is status, not scope.** Inserting a phase there requires a seed update first;
  reorder is fine.
- **Don't fight hooks.** If an ECC hook blocks you, investigate before disabling.

---

## Escalation rules

Stop and surface to creator if:
- A non-negotiable from `seed/creator-vision.md` would be violated
- An ECC agent produces a plan that contradicts the seed
- A context7 result shows a library API is incompatible with the phase plan
- An ECC hook is blocking progress and the cause is unclear
- Three consecutive dev service restart failures
- Serena's `find_referencing_symbols` reveals a refactor is materially larger than the plan assumed
