## Status legend

- `pending` — phase exists in the seed but has no plan file yet
- `in-progress` — `/prp-plan` has produced a plan; `/prp-implement` is running, paused, or in review
- `done` — `/prp-implement` finished, report committed, single-commit-per-phase merged, gates green
- `blocked` — see Notes; `/prp-plan` cannot proceed without a human decision

## How this file is used

1. `/prp-plan` reads the **Next** pointer to know which row to plan. It writes the plan path back into the row and flips status to `in-progress`.
2. `/prp-implement` runs the plan. On the final step it writes the report path, flips status to `done`, and advances the **Next** pointer to the next `pending` row.
3. `/prp-pr` references both the plan and report paths in the PR body.
4. Humans may insert rows (new phases discovered mid-build), reorder rows (priority change), or flip a row to `blocked` with a Note pointing to the open question. Bulk regeneration is forbidden — phase history is part of the audit trail.

---

## Conventions

- Plan files live at `.claude/PRPs/plans/NNNN-phase-N-<slug>.plan.md`. The numeric prefix mirrors the row `#`.
- Report files live at `.claude/PRPs/reports/NNNN-phase-N-<slug>-report.md`.
- Single commit per phase: `feat(phase-N): <scope>`. The commit body lists user-story IDs delivered.
- A phase row is never deleted; if a phase is cancelled, mark it `blocked` and add a Note explaining the cancellation.
